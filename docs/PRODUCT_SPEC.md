## 1. はじめに

### 1.1 目的

本書は、Pokémon Champions の対戦中に相手の構築(パーティ)を予測し、ユーザー自身のパーティに基づいた具体的な対策を提示する Web サービス(以下「本サービス」)の設計を定義する。

### 1.2 サービスのコアバリュー

1. **構築予測** — 対戦中に見えた断片的な情報(ポケモン・技・持ち物など)から、相手が使用しているテンプレ構築を一致度順で予測する
2. **人気度による補正** — 一致度が同じ候補は、実際の使用率・人気が高い構築を優先表示する
3. **パーソナライズされた対策** — 汎用的な攻略情報ではなく、ユーザーが登録した自分のパーティと相手の予測構築を突き合わせ、「今この対戦で何をすべきか」を提案する

### 1.3 想定ユーザー

| ユーザー層 | 利用シーン |
|---|---|
| ランクマッチ中級者 | 選出画面(チームプレビュー)の90秒間に相手構築を予測して選出を決める |
| 初心者 | 対戦後の振り返り・構築の学習 |
| 上級者 | メタ把握・遭遇報告によるデータ貢献 |

### 1.4 用語定義

| 用語 | 意味 |
|---|---|
| テンプレ構築 | 攻略サイト等で広く知られている定番パーティ構成 |
| 一致度 | 判明した相手情報とテンプレ構築の合致度合い(0〜100%) |
| 人気度 | 構築の使用率・注目度を表す複合指標 |
| 遭遇報告 | ユーザーが実際に対戦で遭遇した構築の報告 |
| 選出 | 6体のパーティから対戦に出す3体(ルールによる)を選ぶこと |
| セッション | 1回の対戦に対応する入力・予測の単位 |

### 1.5 スコープ外(v1時点)

- スクリーンショット/映像からの自動認識
- 音声入力
- 対戦の勝率予測(統計データ蓄積後に対応)
- ゲームクライアントとの直接連携(規約上不可の前提)

## 2. システム全体構成

### 2.1 アーキテクチャ概要

```text
┌─────────────────────────────────────────────┐
│                クライアント                    │
│   Webアプリ (SPA / PWA・スマホ最適化)          │
└──────────────┬──────────────────────────────┘
               │ HTTPS (REST + WebSocket)
┌──────────────▼──────────────────────────────┐
│              APIサーバー                      │
│  ・認証 / パーティ管理 / セッション管理         │
│  ・一致度計算エンジン                          │
│  ・相性判定エンジン                            │
│  ・人気度スコア集計(バッチ)                    │
└───────┬──────────────┬──────────────────────┘
        │              │
┌───────▼──────┐  ┌────▼─────────────────────┐
│  PostgreSQL   │  │  LLM API (説明文生成)      │
│  (主データ)    │  │  ※計算結果の言語化のみ      │
└───────────────┘  └──────────────────────────┘
        │
┌───────▼──────┐
│  Redis        │
│  (セッション/  │
│   候補キャッシュ)│
└───────────────┘
```

### 2.2 設計方針

- **判定ロジックはルールベース+DB** — 一致度・相性はすべて決定的なアルゴリズムで計算し、再現性とデバッグ容易性を確保する
- **LLMは説明役に限定** — 「なぜこのポケモンがおすすめか」の自然文生成のみに使用。判定結果そのものはLLMに依存しない
- **リアルタイム性** — 相手情報の入力ごとに候補を即時再計算。対象データは高々数百構築のため、全件スコアリングでも十分高速(後述)
- **スマホファースト** — 対戦中に片手で入力できるUIを最優先

### 2.3 処理フロー(対戦中)

```text
[相手情報を1件入力]
   ↓
POST /sessions/{id}/observations
   ↓
一致度計算エンジン:全テンプレ構築を再スコアリング
   ↓
上位N件(既定3件)を一致度→人気度→遭遇数→更新日でソート
   ↓
レスポンス返却(100ms以内目標)
   ↓
[ユーザーが候補構築を選択]
   ↓
相性判定エンジン:自パーティ × 予測構築 の相性マトリクス計算
   ↓
おすすめ選出・警戒技を算出
   ↓
LLMで理由文を生成(非同期・キャッシュ)
```

## 3. 技術スタック

| レイヤー | 技術 | 選定理由 |
|---|---|---|
| フロントエンド | React + TypeScript + Vite | SPA、型安全、開発速度 |
| UI | Tailwind CSS | スマホ最適化しやすい |
| 状態管理 | TanStack Query + Zustand | サーバー状態と入力状態の分離 |
| バックエンド | Node.js (NestJS) + TypeScript | フロントと型共有(zodスキーマ共通化) |
| DB | PostgreSQL 16 | JSONB活用、全文検索、信頼性 |
| キャッシュ | Redis | セッション・候補キャッシュ |
| LLM | Anthropic API (Claude) | 理由文・立ち回り文の生成 |
| 認証 | メール+パスワード / OAuth (Google) | 参入障壁を下げる |
| インフラ | 任意のPaaS(Render/Fly.io等)→将来AWS | MVPは低コスト運用 |
| ゲームマスタデータ | PokéAPI等を初期投入+自前管理 | Champions固有要素(メガ等)は手動補完 |

> 注: バックエンドは Python (FastAPI) でも可。チームのスキルセットに応じて選択する。本書では TypeScript 前提で記載する。

## 7. 一致度計算エンジン設計

### 7.1 基本方針

- 対象: 現行シーズン・ルールの `published` な全テンプレ構築(想定 数十〜数百件)
- 全件スコアリング方式。件数が少ないため事前絞り込み不要。1構築あたりの計算はO(観測数×構築内要素数)で軽量
- 出力: 各構築の **一致度(0〜100%)** と一致/不一致の内訳

### 7.2 スコアリング仕様

観測ごとに加減点し、最後に正規化する。

#### 配点表(初期値・設定でチューニング可能)

| 観測 | 条件 | 点数 |
|---|---|---|
| ポケモン一致 | 観測ポケモンが構築に存在 | +10 × usage_rate |
| 技一致 | 観測技がそのポケモンの技リストに存在 | +15 × adoption_rate |
| 持ち物一致 | 定番持ち物と一致 | +15(代替持ち物は+8) |
| 特性一致 | 一致 | +8 |
| 先発位置一致 | default_leads の先発と一致 | +6 |
| メガ枠一致 | メガシンカ観測が構築のメガ枠と一致 | +12 |
| ポケモン不一致 | 観測ポケモンが構築に存在しない | −20 |
| 技矛盾 | 構築のそのポケモンにない技を観測 | −12 |
| 持ち物矛盾 | 定番・代替のいずれとも不一致 | −12 |
| 特性矛盾 | 不一致 | −8 |
| メガ矛盾 | 構築にないメガシンカを観測 | −25 |

#### 正規化

```text
raw_score  = Σ(加点) + Σ(減点)
max_score  = その時点の観測列がすべて完全一致した場合の理論最大点
match_rate = clamp(raw_score / max_score, 0, 1) × 100
```

- 観測が増えるほど max_score が増え、一致度が安定する
- 減点により raw_score が負になる場合は 0% とする
- **除外条件:** ポケモン不一致が3体以上、またはメガ矛盾が発生した構築は候補から除外(表示しない)

### 7.3 ソート仕様

```text
ORDER BY
  match_rate      DESC,   -- 1. 一致度
  popularity_rank ASC,    -- 2. 人気度(high→mid→low、将来はscore DESC)
  encounter_count DESC,   -- 3. 遭遇報告数
  updated_at      DESC    -- 4. データの新しさ
LIMIT 3(設定可能)
```

### 7.4 表示要素の算出

| 表示 | 算出方法 |
|---|---|
| 一致した情報 | 加点対象となった観測のリスト |
| 残りの可能性が高いポケモン | 構築内で未観測のポケモンを usage_rate 降順 |
| 警戒すべき技 | 未観測ポケモン+観測済みポケモンの未観測技のうち、`tags` に `setup`/`hazard`/`screen`/`priority` を含む技、および threat_notes 記載技 |

### 7.5 擬似コード

```typescript
function scoreArchetype(archetype: Archetype, obs: Observation[]): ScoredCandidate {
  let raw = 0, max = 0;
  const matched: MatchDetail[] = [];
  let missCount = 0, megaConflict = false;

  for (const o of obs.filter(o => !o.isRevoked)) {
    const w = WEIGHTS[o.kind];
    max += w.hit;
    const hit = matchObservation(archetype, o); // 一致判定
    if (hit.matched) { raw += hit.points; matched.push(hit); }
    else {
      raw -= w.penalty;
      if (o.kind === "pokemon") missCount++;
      if (o.kind === "mega") megaConflict = true;
    }
  }
  const excluded = missCount >= 3 || megaConflict;
  return {
    archetypeId: archetype.id,
    matchRate: Math.max(0, Math.min(1, raw / max)) * 100,
    matched, excluded,
  };
}
```

### 7.6 呼称ルール

統計的裏付け(遭遇報告等)が十分に蓄積されるまで、UI上の表記は「確率」ではなく **「一致度」** で統一する。

## 9. 相性判定エンジン設計

### 9.1 入力と出力

- **入力:** 自パーティ(6体・全情報)× 予測構築(相手6体。観測済み情報は実測を優先、未観測はテンプレ既定値で補完)
- **出力:**
  - 相手各ポケモンに対する自ポケモンのおすすめ順位+スコア内訳
  - パーティ全体のおすすめ選出(先発/控え/エース)
  - 警戒技・警戒ムーブ
  - 基本立ち回り方針

### 9.2 1対1相性スコア

自ポケモン A vs 相手ポケモン B について:

| 評価軸 | 内容 | 配点(初期値) |
|---|---|---|
| 攻撃相性 | Aの技でBの弱点を突けるか(タイプ相性×技威力×攻撃実数値) | 0〜30 |
| 防御相性 | Bの想定技をAが半減/無効にできるか(特性込み) | 0〜30 |
| 素早さ関係 | 実数値比較(スカーフ等の持ち物補正込み)。上を取れる=+、抜かれる=− | −10〜+15 |
| ダメージ概算 | 簡易ダメージ計算による確定数比較(何発で倒せる/倒される) | −15〜+15 |
| 先制技 | Aが有効な先制技を持つ | +5 |
| 状態異常耐性 | Bの状態異常技(あくび等)への耐性 | 0〜5 |
| 積み対応 | Bが積み技持ちの場合、A側に対抗手段(挑発・ほえる・天然等)があるか | −10〜+10 |

```text
matchup_score(A, B) = Σ(各評価軸) を −100〜+100 に正規化
  +50以上: 有利   +10〜+50: やや有利
  −10〜+10: 五分  −50〜−10: やや不利  −50以下: 不利
```

- 相手の技が観測済みならその技で計算、未観測ならテンプレの採用率上位4技で計算
- 観測情報が更新されるたびに再計算する

### 9.3 ダメージ概算仕様

正式なダメージ計算式の簡易版を使用する:

```text
damage ≒ (22 × 威力 × 攻撃実数値 / 防御実数値) / 50 + 2
         × タイプ一致(1.5) × タイプ相性 × 持ち物/特性補正
確定数 = ceil(相手HP実数値 / damage下限)
```

乱数・急所は考慮しない。テラスタイプが登録されている場合はテラス後の相性も併記する。

### 9.4 選出提案アルゴリズム

1. 自6体 × 相手6体 の相性マトリクス(36セル)を計算
2. 各自ポケモンの評価値 = 「有利を取れる相手の数と質」+「不利対面の少なさ」+「相手の主軸(role=sweeper/エース)への回答になるか」を加重合計
3. 上位から3体を選出候補とする。ただし以下の制約を適用:
   - 相手のエース級に対する回答が最低1体含まれること
   - 起点回避手段(挑発・自主退場・ほえる等)の有無を評価
4. 先発: 相手の default_leads(テンプレの基本先発)に対して最も有利なポケモン
5. エース: 評価値が高く、積み技/高火力を持つポケモン
6. 控え: 残り1枠。先発とエースが苦手な相手をカバーするポケモン(補完重視)

### 9.5 警戒技・立ち回りの生成

- **警戒技:** 相手構築内の技のうち `tags` が `setup` / `hazard` / `screen` / `priority` / `status` のもの+各 `threat_notes` を優先度順に列挙
- **基本方針:** テンプレの `playstyle_notes` をベースに、自選出との組み合わせで注意点を追加(例: 壁構築なら「壁ターンを枯らす」「挑発持ちを先発に」)。文章化はLLMが担当(§12)
